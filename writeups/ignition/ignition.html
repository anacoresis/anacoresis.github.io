HTB Machine: Ignition

OS · Linux Difficulty · Very Easy (Tier 0) July 2026 Target · 10.129.1.27

Step 1 · Reconnaissance & Scanning

Run nmap to determine the version of the service running on port 80:

sudo nmap -sV -sC 10.129.1.27

Key findings

  • Target IP: 10.129.1.27
  • Port 80 (HTTP): Open.
  • Web server: nginx 1.14.2.

Step 2 · Investigating the Redirect

I initially tried to navigate to the web page at http://10.129.1.27, but it resolved to http://ignition.htb/ and returned the error: "Hmm. We're having trouble finding that site."

The network tab did not display an explicit error code in the request headers. Checking the hint revealed that curl was the right tool to investigate:

curl -v http://10.129.1.27

This surfaced the redirection status code we were looking for: 302 Found.

Step 3 · Virtual Host Routing Fix

To understand why the browser failed to load the page, I researched the issue. An IP address is just a physical location on a network. A single Nginx server at one IP can host dozens of completely different websites simultaneously using virtual hosting.

When typing the raw IP into the browser, Nginx sees Host: 10.129.1.27. Because it does not recognize that IP as a specific website configuration, it fails to route the request. Nginx needs to see a specific domain name (like Host: ignition.htb) in the HTTP header to route traffic to the Magento application.

To fix this, I edited the local hosts file:

sudo vim /etc/hosts

I added the line 10.129.1.27 ignition.htb, saved, and exited. Navigating to the web application then worked.

Step 4 · Web Directory Brute-Forcing

Directory busting was performed with gobuster to find hidden paths:

gobuster dir -u http://ignition.htb -w /usr/share/wordlists/dirb/common.txt -t 30

The -t 30 flag sets threads, meaning the tool checks 30 words from the wordlist at a time. This yielded a few results, including the /admin page.

Step 5 · Admin Login & Flag Retrieval

The platform requires an administrative password. Magento defends against brute forcing — too many incorrect attempts lock the user out for a couple of minutes — so this required manual trial and error.

The process involved researching common default administrative passwords. The correct password was entry number 48 on the list: qwerty123.

Note

I did not manually guess all 48 passwords; because this is an older machine, I looked up the standard password list reference. This mimics real-world scenarios where known password lists are prioritized based on platform defaults.
